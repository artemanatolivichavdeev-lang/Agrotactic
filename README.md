README.txt — Проект сайта Agrotaktik (Enterprise)

Дата: 2026-02-17

──────────────────────────────────────────────────────────────────────────────
1) Что это за сайт
──────────────────────────────────────────────────────────────────────────────
Это корпоративный сайт‑витрина (showcase site) продукта «Агротактик» для enterprise‑аудитории
агрохолдингов. Сайт проектируется и собирается как “доказательный” инструмент для прохождения
коллективной покупки и внутренних согласований пилота/внедрения.

Публикация выполняется в Tilda (через HEAD и блоки T123). Разработка ведётся в Google Antigravity,
но вся “истина” по коду/артефактам хранится в одной папке проекта: /tilda (без дублирования версий).

Язык UI‑контента сайта: русский.

──────────────────────────────────────────────────────────────────────────────
2) Кому сайт продаёт и зачем (Advance JTBD)
──────────────────────────────────────────────────────────────────────────────
Ключевой JTBD посетителя:
«Продвинуть внутреннее решение о пилоте/внедрении» в условиях коллективной покупки, где участвуют:
- COO / директор по производству / операционный блок
- Главный инженер / эксплуатация / парк
- CIO/CTO / цифровизация / IT
- ИБ (информационная безопасность)
- Юристы / комплаенс
- Закупка / procurement

Задача Home в терминах progress:
1) Frame (8–12 секунд): сразу понять, что это “операционный контур / командный центр”, а не «ещё один софт».
2) De‑risk: увидеть безопасность/ИБ/комплаенс/документность как артефакты (briefs, схемы, пакеты), а не обещания.
3) Prove: понять, как доказываем результат на пилоте (метрики, критерии приёмки, выходные артефакты).
4) Route: получить “маршрут по роли” и следующий шаг: демо/техсессия/пакет документов/пилот.

──────────────────────────────────────────────────────────────────────────────
3) Позиционирование и тон бренда
──────────────────────────────────────────────────────────────────────────────
Архетипы бренда: Правитель + Мудрец + Хранитель.
Тон: строгий, инженерный, доказательный, “procurement‑ready”.

Запрещено (по стилю):
- «AI‑магия», обещания без артефактов, кислотные градиенты, «вау‑видео ради вау».
Разрешено:
- функциональные микровзаимодействия (drawer, tabs, hotspots, accordion), строгая сетка,
табличность, чипы, лейблы, тонкие линии, “mission control” эстетика.

──────────────────────────────────────────────────────────────────────────────
4) Что показывает продукт (смысловая модель)
──────────────────────────────────────────────────────────────────────────────
«Агротактик» — это операционный контур / оркестратор:
- диспетчеризация и алерты (очередь отклонений, приоритеты, ответственность)
- план‑факт и KPI‑наблюдаемость в реальном времени
- интеграции поверх существующих систем (ERP/1С, телематика/датчики, BI, API)
- журнал действий / audit trail (кто‑что‑когда‑почему)
- комплаенс и отчётность как часть процесса (включая «Меркурий»)
- управляемая “лестница автономии” A2 → A3 → A4 (поэтапно, без скачков и рисков)

──────────────────────────────────────────────────────────────────────────────
5) Информационная архитектура Home (цепочка доказательств)
──────────────────────────────────────────────────────────────────────────────
Home построена в логике: Frame → De‑risk → Prove → Procure → Switch.

Ключевые элементы первого экрана (Hero):
- H1 + субхед: сразу “командный центр/контур управления”
- 2 CTA: «Запросить демо/техсессию» и «Получить пакет документов»
- Role Switcher (3 роли): Руководителю / IT+ИБ / Инженеру
  (меняет аргументы и вторичный CTA под роль)
- Proof Overlay на мокапе (hotspots + tooltips)
- Under‑hero strip:
  - Trust badges (Safety/Security/Integration/Compliance/Procurement ready)
  - Mini data‑flow “ERP/1С → Orchestrator → Command Center → Compliance/Reports”
- Proof Drawer (Trust Pack): боковая панель/боттом‑шит с документами:
  One‑pager, Pilot Blueprint, Architecture brief, Security brief, Compliance brief
  (+ потенциально Safety brief, если будет добавлен как 6‑й документ)

Ниже Hero (строгий порядок секций):
- Операционная модель (схема 5 узлов) — “что контролируем / что измеряем / чем подтверждаем”
- 4 слоя платформы (Orchestrator / Command Center / Integration Hub / Compliance Suite)
- Лестница автономии A2→A3→A4 (stepper + требования чипами)
- Командный центр (сценарии демо: «отклонение → алерт → решение»)
- Интеграции (ERP/1С, телематика, BI, API) + “Self‑check 2 минуты”
- Комплаенс (вкл. «Меркурий»)
- Безопасность + ИБ (Trust Pack)
- Пилот 6–8 недель (метрики, приёмка, выходные артефакты)
- Procurement Pack (для закупки)
- Финальный CTA (перевод в действие)

──────────────────────────────────────────────────────────────────────────────
6) Конверсия и формы
──────────────────────────────────────────────────────────────────────────────
Основная конверсия: заявка на демо/техсессию.
Рекомендуемый режим: нативная форма Tilda (без кода).

Важное правило: CTA‑кнопки должны работать без JS‑скролла — через якоря Tilda вида #recXXXXXX
(реальные ID блоков формы и контактов фиксируются в /tilda/docs/TILDA_ANCHORS.md).

──────────────────────────────────────────────────────────────────────────────
7) Техническая реализация (Antigravity → Tilda)
──────────────────────────────────────────────────────────────────────────────
Публикация:
- HEAD: Site Settings → More → HTML code for the HEAD section
- T123: блок “Embed HTML Code” на странице

Ограничения Tilda:
- T123 выполняется только на опубликованной странице (в preview может не исполняться)
- после изменений в HEAD требуется “Publish all pages”
- серверный код (PHP и т.п.) на странице невозможен; интеграции — через Webhook на внешнем endpoint

Стек (вставляемый код):
- HTML5 + CSS3 + JS (ES6+), без React/Vue/Bootstrap
- строгая изоляция стилей:
  - контейнер .x-site
  - все классы x-*
  - без глобальных селекторов body/h1/a без префикса .x-site

8) Структура проекта на диске (важное)
──────────────────────────────────────────────────────────────────────────────
Режим TILDA‑ONLY: вся работа ведётся в /tilda.

Рекомендуемая структура (внутри репозитория/папки проекта):
/refs/                          — референсы (скриншоты/вдохновение)
/tilda/
  /head/                        — содержимое для HEAD (site-head.html и др.)
  /t123/                        — сниппеты для T123 (home/home.html и др.)
  /assets/                      — ассеты (изображения/иконки) для загрузки в Tilda
  /docs/                        — рабочие документы процесса (WORKFLOW, CHECKLIST, ANCHORS)
  /releases/                    — снимки релизов (копии того, что вставляли в Tilda)

Файлы процесса:
- /tilda/docs/WORKFLOW.md       — правила работы (Hero-first, Tilda constraints, порядок вставки)
- /tilda/docs/COPY_TO_TILDA.md  — шпаргалка “что копировать куда” (HEAD vs T123) + порядок Publish
- /tilda/docs/QA_CHECKLIST.md   — быстрый pre‑launch чеклист
- /tilda/docs/TILDA_ANCHORS.md  — реальные #rec... якоря формы и контактов
- /tilda/docs/CHANGELOG.md      — изменения по версиям

──────────────────────────────────────────────────────────────────────────────
9) Definition of Done (кратко)
──────────────────────────────────────────────────────────────────────────────
Hero готов, если:
- визуально и по тону соответствует “industrial tactical / mission control”
- адаптив 390/768/1200/1440 без поломок
- role switcher реально меняет контент
- drawer открывается из header/бейджей/CTA
- hotspots работают на desktop (hover) и mobile (tap)
- CTA ведут на реальные #rec... якоря (без JS)
- на опубликованной странице нет ошибок в консоли

Сайт готов к передаче/запуску, если:
- пройдён /tilda/docs/QA_CHECKLIST.md
- формы доставляют лиды (3 теста)
- после изменений в HEAD выполнен publish all pages
- сохранён релизный снимок в /tilda/releases/YYYY-MM-DD_vX/

──────────────────────────────────────────────────────────────────────────────
10) Примечания по рабочему процессу
──────────────────────────────────────────────────────────────────────────────
- “Single Source of Truth = код в /tilda”. Tilda — это канал публикации.
- Не вести две версии проекта (это причина сбоев Antigravity в прошлых итерациях).
- Любые документы (briefs/PDF) сначала готовятся как файлы/ссылки, затем загружаются в Tilda,
  и только после этого подставляются реальные URL в кнопки скачивания.
